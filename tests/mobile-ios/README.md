# Mobile iOS Tests — Universal Downloader by CT4nk3r

XCTest + XCUITest coverage for the iOS host app and Share Extension. Tests
target a mocked backend at `http://127.0.0.1:8787` (see `apps/services/api`
contract tests for spinning one up).

## Test files

| File                                                                      | Kind     | Purpose                                              |
|---------------------------------------------------------------------------|----------|------------------------------------------------------|
| `apps/mobile/ios/UniversalDownloaderTests/AppLaunchTests.swift`           | XCTest   | App boots, RN root view renders.                     |
| `apps/mobile/ios/UniversalDownloaderUITests/QueueFlowUITests.swift`       | XCUITest | Paste URL → pick preset → enqueue → progress shown.  |
| `apps/mobile/ios/UniversalDownloaderUITests/ShareExtensionUITests.swift`  | XCUITest | Share extension `.appex` is embedded (smoke).        |

## Required Xcode targets (NOT auto-added)

The Xcode project (`UniversalDownloader.xcodeproj`) is generated by
`pod install`. Agent J2.4 did **not** edit `project.pbxproj`. Before tests
can be discovered/run, the following targets must exist in the workspace:

1. **`UniversalDownloaderTests`** — Unit Test Bundle
   - Host application: `UniversalDownloader`
   - Sources: `apps/mobile/ios/UniversalDownloaderTests/AppLaunchTests.swift`
   - Info.plist: `apps/mobile/ios/UniversalDownloaderTests/Info.plist`
   - Bundle id: `com.ct4nk3r.universaldownloader.tests`

2. **`UniversalDownloaderUITests`** — UI Testing Bundle
   - Target application: `UniversalDownloader`
   - Sources:
     - `apps/mobile/ios/UniversalDownloaderUITests/QueueFlowUITests.swift`
     - `apps/mobile/ios/UniversalDownloaderUITests/ShareExtensionUITests.swift`
   - Info.plist: `apps/mobile/ios/UniversalDownloaderUITests/Info.plist`
   - Bundle id: `com.ct4nk3r.universaldownloader.uitests`

3. Add both test targets to the **`UniversalDownloader`** scheme's *Test*
   action so `xcodebuild test` picks them up automatically.

Adding these is a one-time manual step (or a `xcodegen`/`tuist` task) — once
done, the files in this PR are all that's needed.

## Running

From the repo root (macOS host with Xcode 15+):

```bash
# Install JS deps + native pods (J2.4 did NOT run these — do once locally)
cd apps/mobile
npm install
cd ios && pod install && cd ..

# Boot the mock API the tests target
PORT=8787 npm --prefix ../services/api run mock &

# Unit tests only
xcodebuild test \
  -workspace ios/UniversalDownloader.xcworkspace \
  -scheme UniversalDownloader \
  -destination 'platform=iOS Simulator,name=iPhone 15,OS=latest' \
  -only-testing:UniversalDownloaderTests

# UI tests only
xcodebuild test \
  -workspace ios/UniversalDownloader.xcworkspace \
  -scheme UniversalDownloader \
  -destination 'platform=iOS Simulator,name=iPhone 15,OS=latest' \
  -only-testing:UniversalDownloaderUITests

# Everything
xcodebuild test \
  -workspace ios/UniversalDownloader.xcworkspace \
  -scheme UniversalDownloader \
  -destination 'platform=iOS Simulator,name=iPhone 15,OS=latest'
```

## Test environment contract

UI tests pass these via `XCUIApplication.launchEnvironment`:

| Var               | Value                    | Meaning                                         |
|-------------------|--------------------------|-------------------------------------------------|
| `UD_API_BASE_URL` | `http://127.0.0.1:8787`  | Mock backend base URL the JS layer should use.  |
| `UD_TEST_MODE`    | `1`                      | Disables analytics, animations, background polling. |

The RN side should read these in `apps/mobile/src/lib/api-context.tsx` and
fall back to defaults when absent so the same code paths work in production
builds.

## TestID conventions referenced

These are the accessibility identifiers `QueueFlowUITests` looks for. Keep
them stable across UI refactors:

- `url-input` — TextInput on Home
- `paste-btn` — Paste-from-clipboard button on Home
- `download-btn` — Enqueue button on Home
- `preset-best` / `preset-1080p` / `preset-720p` / `preset-audio` — preset chips (any one)
- `queue-screen` — Queue tab root view
- `job-row-<id>` — individual job row in Queue list
